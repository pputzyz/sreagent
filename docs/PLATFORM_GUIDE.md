# SREAgent 平台操作手册

> v4.44.5 | 2026-05-27

## 平台概览

SREAgent 是一个完整的 SRE 监控告警值班平台，包含 3 大应用模块、62 个数据模型、350+ API 端点。

### 三大应用

| 应用 | 路由前缀 | 定位 |
|------|----------|------|
| **On-Call 值班** | `/oncall` | 事件响应、排班、通知分发、协作空间 |
| **Alert 告警** | `/alert` | 告警规则、事件管理、数据源、数据探索 |
| **Platform 平台** | `/platform` | 用户/团队、系统设置、AI、任务执行 |

### 角色权限

| 角色 | 能力 |
|------|------|
| **admin** | 全部功能，包括用户管理、SSO、审计日志 |
| **team_lead** | 创建/编辑规则、排班、通知配置；不能管理用户 |
| **member** | 确认/分配事件、查看规则和排班、创建任务 |
| **viewer** | 只读查看 |

## 模块一：告警全生命周期

### 1.1 数据源配置

**路径**: Alert → 数据源

**作用**: 连接 Prometheus、VictoriaMetrics、VLogs、Zabbix、Elasticsearch 等数据源，告警规则和数据查询都依赖数据源。

**操作步骤**:
1. 进入 Alert → 数据源
2. 点击"添加数据源"
3. 选择类型（Prometheus / VictoriaMetrics / Elasticsearch 等）
4. 填写名称、URL、认证信息
5. 点击"测试连接"验证
6. 保存

**验证**: 数据源列表显示绿色健康状态。

### 1.2 告警规则

**路径**: Alert → 告警规则

**作用**: 定义告警条件。规则绑定数据源，通过 PromQL/MetricsQL 表达式定义阈值。

**操作步骤**:
1. 进入 Alert → 告警规则
2. 点击"创建规则"
3. 配置：
   - **基本信息**: 名称、严重等级 (P0-P4)
   - **数据源**: 选择已配置的数据源
   - **表达式**: PromQL 查询，如 `up == 0`
   - **持续时间**: 条件持续多久后触发 (for duration)
   - **标签**: 附加标签用于路由和分组
   - **通知**: 选择通知策略
4. 保存

**规则状态**: draft(草稿) / active(生效) / disabled(禁用)

**验证**: 规则列表显示新规则，状态为 active。

### 1.3 告警事件

**路径**: Alert → 活跃告警 / 告警历史

**作用**: 展示触发的告警事件，支持全生命周期管理。

**事件状态流转**:
```
firing(触发) → acknowledged(确认) → assigned(分配) → resolved(恢复) → closed(关闭)
```

**操作**:
- **确认**: 点击事件 → 确认，标记已有人处理
- **分配**: 指派给具体处理人
- **恢复**: 条件不再满足时自动恢复，或手动恢复
- **关闭**: 处理完毕关闭
- **导出**: 支持 CSV 导出

**验证**: 创建一个测试规则触发后，事件出现在活跃告警列表。

### 1.4 静默与抑制

**路径**: Alert → 静默规则 / 抑制规则

**静默规则**: 在指定时间窗口内屏蔽匹配的告警（如维护窗口）。支持：
- 一次性时间窗口
- 周期性静默（如每周六 2:00-4:00）
- 命中预览：创建前查看哪些规则会被静默

**抑制规则**: Alertmanager 风格，当 source 告警触发时，自动抑制 target 告警（如核心交换机 down 时抑制其下游告警）。

### 1.5 事件管道

**路径**: Alert → 事件管道

**作用**: 可编程告警处理链，在告警评估后、通知前执行。支持处理器：
- `relabel`: 修改告警标签
- `callback`: 调用外部 webhook
- `event_drop`: 丢弃匹配的告警
- `ai_summary`: AI 生成摘要
- `logic.if` / `logic.switch`: 条件分支

### 1.6 录制规则

**路径**: Alert → 录制规则

**作用**: 定期执行 PromQL 查询并将结果写回数据源，用于预计算昂贵查询。

## 模块二：On-Call 值班管理

### 2.1 协作空间 (Channel)

**路径**: On-Call → 协作空间

**作用**: 按团队或业务线组织告警。每个空间有独立的：
- 告警事件视图
- 排除规则（过滤噪音告警）
- 分派策略（路由到不同通知渠道）
- 降噪配置

### 2.2 值班排班

**路径**: On-Call → 排班管理

**作用**: 定义值班轮转，确保告警 24/7 有人响应。

**操作步骤**:
1. 创建排班计划
2. 选择轮转周期（每日/每周/自定义）
3. 设置交接时间
4. 添加参与人（按顺序轮转）
5. 可选：设置替班（临时替换）

**验证**: 排班详情页显示当前值班人。

### 2.3 升级策略

**路径**: On-Call → 配置中心 → 升级策略

**作用**: 定义告警未处理时的升级链。例如：
- Step 1: 通知值班人（立即）
- Step 2: 通知团队 Lead（5 分钟后）
- Step 3: 通知所有团队成员（15 分钟后）

**升级目标类型**:
- `user`: 直接通知指定用户
- `team`: 通知整个团队
- `schedule`: 通知当前值班人

### 2.4 通知配置

**路径**: On-Call → 通知中心

**四大功能**:

| 菜单 | 作用 |
|------|------|
| **通知策略** | 定义哪些告警触发哪些通知渠道 |
| **通知渠道** | 配置 17 种渠道：飞书、钉钉、企业微信、Slack、邮件、短信等 |
| **消息模板** | 自定义通知消息格式 |
| **订阅规则** | 用户自定义订阅感兴趣的告警 |

### 2.5 集成中心

**路径**: On-Call → 集成

**作用**: 接入外部告警源（Alertmanager、Grafana、自定义 Webhook），通过路由规则分发到不同协作空间。

### 2.6 故障管理

**路径**: On-Call → 故障

**作用**: 故障全生命周期管理：创建 → 响应 → 处理 → 恢复 → 复盘。

**关联**: 故障自动关联告警事件，支持 AI 生成复盘初稿。

## 模块三：平台管理

### 3.1 用户与团队

**路径**: Platform → 组织

- **成员管理** (admin): 用户 CRUD、角色分配
- **团队管理**: 团队 CRUD、成员添加
- **角色权限**: 查看各角色权限矩阵
- **SSO 配置**: Keycloak OIDC / LDAP / 通用 OAuth2

### 3.2 系统设置

**路径**: Platform → 设置

- **SMTP**: 邮件通知配置
- **飞书机器人**: 飞书 Webhook/Bot 配置
- **AI 配置**: LLM 提供商、MCP 服务器、Skill 管理
- **安全设置**: 登录策略、密码策略
- **联系方式**: 用户联系方式管理

### 3.3 AI 助手

**路径**: AI Agent / Platform → AI 配置

- **AI Agent**: 对话式告警分析、根因诊断、SOP 建议
- **AI 规则生成**: 描述需求自动生成告警规则
- **诊断工作流**: AI 驱动的自动化诊断流程

### 3.4 任务执行

**路径**: Platform → 任务模板 / 任务记录

**作用**: 远程执行自愈脚本。流程：
1. 创建脚本模板（支持 SSH 远程执行）
2. 手动触发或告警自动触发
3. 查看每台主机的执行结果（stdout/stderr/退出码）

### 3.5 其他

- **变更事件**: 记录部署、配置变更，辅助根因分析
- **知识库**: SOP 文档管理
- **标注管理**: 在仪表盘时间线上标注事件
- **巡检**: 定期健康检查

## 核心工作流

### 工作流一：告警全生命周期

```
数据源配置 → 创建告警规则 → 引擎评估 → 触发事件 → 通知分发 → 值班人响应 → 确认/恢复
```

**详细步骤**:
1. Alert → 数据源：添加 Prometheus
2. Alert → 告警规则：创建规则 `up == 0`，绑定数据源
3. 等待引擎评估（默认 15s 周期）
4. Alert → 活跃告警：查看触发的事件
5. 点击事件 → 确认 → 分配给处理人
6. 修复后事件自动恢复或手动恢复

### 工作流二：值班 On-Call

```
创建团队 → 配置排班 → 设置升级策略 → 配置通知渠道 → 告警自动路由到值班人
```

**详细步骤**:
1. Platform → 团队：创建团队、添加成员
2. On-Call → 排班：创建排班计划、设置轮转
3. On-Call → 升级策略：定义升级链（值班人 → Lead → 全员）
4. On-Call → 通知渠道：配置飞书/邮件/钉钉等
5. On-Call → 通知策略：关联告警到通知渠道

### 工作流三：集成外部告警

```
创建集成 → 配置路由规则 → 外部告警接入 → 自动分发到协作空间
```

**详细步骤**:
1. On-Call → 集成：创建集成（获取 Webhook URL）
2. On-Call → 路由规则：定义标签匹配规则
3. 外部系统发送告警到 Webhook URL
4. 告警自动路由到对应协作空间

## 日志查看与验证

### 后端日志

```bash
# 实时查看日志
tail -f logs/sreagent.log | grep -E "level=error|level=warn"

# 查看告警引擎日志
grep "alert-engine" logs/sreagent.log

# 查看通知发送日志
grep "notify" logs/sreagent.log

# 查看升级执行日志
grep "escalation" logs/sreagent.log
```

### 前端调试

1. 打开浏览器 DevTools → Network 标签
2. 观察 API 请求是否返回 200
3. Console 标签查看是否有 JS 错误
4. 特别关注 `usePermissions` 的 `hasPerm` 日志

### API 验证

```bash
# 健康检查
curl http://localhost:8080/healthz

# 登录获取 token
curl -X POST http://localhost:8080/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"your_password"}'

# 使用 token 查询告警规则
curl http://localhost:8080/api/v1/alert-rules \
  -H "Authorization: Bearer YOUR_TOKEN"

# 查询告警事件
curl http://localhost:8080/api/v1/alert-events \
  -H "Authorization: Bearer YOUR_TOKEN"
```

## 已知问题与限制

1. **NotifyRule vs DispatchPolicy**: 两套并行的通知机制，NotifyRule 使用 Pipeline，DispatchPolicy 使用升级策略。未来计划统一。
2. **告警 v1/v2 共存**: 部分模块使用 v1 API，部分使用 v2，需要逐步迁移。
3. **测试覆盖**: 大部分模块单元测试覆盖率为 0%，核心引擎约 40%。
