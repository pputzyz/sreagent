# 国际化（i18n）架构决策

> 关联 ADR-16 | 最后更新：2026-06-13

本文记录 SREAgent 的国际化分层原则：**翻译归属取决于这段文案"给谁看、走不走前端"**，而不是"统一放前端"或"统一放后端"。

---

## 核心原则

| 文案类型 | 归属 | 做法 | 理由 |
|----------|------|------|------|
| **UI 文案**（按钮/标签/菜单/表单/提示） | **前端** | vue-i18n（`web/src/i18n/`，zh-CN / en） | 标准做法；UI 呈现层就该拥有 UI 文案。**不应**搬到后端。 |
| **API 错误信息** | **前端为主，后端兜底** | 后端返回稳定 `code`，前端 `errorCodeMap`（`web/src/api/request.ts`）映射到 i18n key 渲染 | SPA 推荐模式：后端给机器可读码，前端负责本地化呈现。 |
| **出站文案**（飞书卡片/邮件/AI 输出/报告正文） | **后端** | 需按**收件人**语言渲染（per-recipient locale） | 这些内容**绕开前端**，由后端直接推给用户，前端 i18n 永远碰不到。 |

---

## 为什么不把翻译整体搬到后端

1. 前端已有成熟 vue-i18n 体系（zh/en 键完全对齐、绝大多数组件接入），UI 文案留前端是正确架构。
2. API 错误用"**码 + 前端翻译**"已是最佳实践：后端 `AppError.Code` 稳定，前端按码渲染本地化文本（`errorCodeMap`）。这一层前端已做对。
3. 因此**后端的错误消息本地化对 Web UI 而言是冗余的**——它与前端 `errorCodeMap` 重复。

## 后端错误消息本地化的真实定位（已实现，低优先级）

- 位置：`internal/pkg/i18n`（`Negotiate` 语言协商 + 以"规范英文消息串"为键的译表 `LocalizeMessage`），接入 `handler.Error`。
- 语言来源：请求头 `X-Lang`（前端 `request.ts` 拦截器同步应用内语言切换）或 `Accept-Language`，默认 `en`（不改变既有 API 行为）。
- 价值：让后端**自给自足**，服务于**直接读 message 文本的非 Web API 消费者**（脚本/第三方集成）。对只被本前端调用的接口，加不加都行。
- 边界：仅覆盖**结构化错误响应**。`WithMessage` 自定义文案与未知串原样透传，**不会误译**。

## 后端唯一"非做不可"的 i18n（待立项）

真正只能在后端解决、且前端无法替代的，是**出站文案**：

- 飞书卡片 / 邮件通知正文
- AI / Agent 输出、巡检与报告正文
- 服务层散落的面向用户的 `fmt.Errorf("中文…")`（若它们会进入通知/卡片）

**难点**：这些要按**收件人**的语言偏好渲染（per-recipient locale），而不是按某个 HTTP 请求的 `Accept-Language`——因为一条告警可能同时推送给语言偏好不同的多个接收人。

### 已实现：AI 响应按请求用户语言（per-recipient）

- 语言来源：`model.UserPreference.Language`（`zh-CN | en`，已有字段）。
- `AgentService.resolveOutputLang(ctx, userID)` 解析请求用户的语言偏好（缺失/未知/匿名→默认 `zh-CN`，永不报错）；`SetUserPreferenceService` 两阶段注入（wire.go）。
- AI Agent 面向用户的最终回答 system prompt 按该语言切换中/英（`service/ai_agent.go` RunAgent）。规划/失败判断等内部 prompt 不本地化（输出不直接面向用户）。
- 这是 per-recipient 能干净落地的场景：AI 对话有**单一**请求用户。

### 已实现：群播卡片按渠道语言（channel-level）

- **群播卡片无法 per-recipient**（一条飞书群消息发给多个语言不同的人），故语言挂在**渠道（NotifyMedia）维度**。
- `pkg/lark.cardLabelsFor(lang)` 提供卡片静态标签目录（zh-CN 默认 | en）；`BuildWebhookCard(..., lang)` 据此渲染状态/字段/AI 分析/按钮等标签。
- `NotifyMedia` 的 lark_webhook / feishu_webhook / feishu_card / feishu_app 配置新增 `language` 字段；前端 `Media.vue` 提供"卡片语言"下拉。
- 仅群播 webhook 卡片本地化；DM/逐用户卡片走 per-recipient（用户偏好）路径。

### 已实现：消息模板多语言变体（覆盖所有渠道的正文）

- `MessageTemplate.ContentEN` 英文变体；`RenderTemplateLang(ctx,id,data,lang)` 按语言选变体，回退 `Content`。
- 渲染入口按**渠道语言**（`mediaLanguage(media)`，解析 media.Config 的 language）选择变体：notify_rule、scheduled_dispatch。
- 因此 email/dingtalk/wecom/slack/discord 等**纯模板驱动**渠道的本地化即"配两份模板正文"，无需改代码。
- 迁移 `000125_message_template_content_en`。

### 仍未做

- 服务层散落中文 `fmt.Errorf`：已确认对中文用户直接显示中文（见上文），无需译表。
- email 主题等极少数中性静态串（`[SREAgent][SEVERITY] name`，无需本地化）。

---

## 决策小结

- 翻译**不应**整体搬到后端；前端 i18n 保留。
- API 错误用"码 + 前端翻译"已足够；后端错误译表作非 Web 消费者的兜底，属锦上添花。
- 后端 per-recipient i18n 的可落地场景是**有单一收件人**的出站内容（AI 对话已实现）；群播内容需按渠道/任务配置语言，是另一套设计。
