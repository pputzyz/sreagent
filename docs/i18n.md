# 国际化（i18n）架构决策

> 关联 ADR-16 | 最后更新：2026-06-13

本文记录 SREAgent 的国际化分层原则：**翻译归属取决于这段文案"给谁看、走不走前端"**，而不是"统一放前端"或"统一放后端"。

---

## 核心原则

| 文案类型 | 归属 | 做法 | 理由 |
|----------|------|------|------|
| **UI 文案**（按钮/标签/菜单/表单/提示） | **前端** | vue-i18n（`web/src/i18n/`，zh-CN / en） | 标准做法；UI 呈现层就该拥有 UI 文案。**不应**搬到后端。 |
| **API 错误信息** | **前端为主，后端兜底** | 后端返回稳定 `code`，前端 `errorCodeMap`（`web/src/api/request.ts`）映射到 i18n key 渲染 | SPA 推荐模式：后端给机器可读码，前端负责本地化呈现。 |
| **出站文案**（飞书卡片/邮件/AI 输出/报告正文） | **后端** | 需按**收件人**语言渲染（per-recipient locale） | 这些内容**绕开前端**，由后端直接推给用户，前端 i18n 永远碰不到。 |

---

## 为什么不把翻译整体搬到后端

1. 前端已有成熟 vue-i18n 体系（zh/en 键完全对齐、绝大多数组件接入），UI 文案留前端是正确架构。
2. API 错误用"**码 + 前端翻译**"已是最佳实践：后端 `AppError.Code` 稳定，前端按码渲染本地化文本（`errorCodeMap`）。这一层前端已做对。
3. 因此**后端的错误消息本地化对 Web UI 而言是冗余的**——它与前端 `errorCodeMap` 重复。

## 后端错误消息本地化的真实定位（已实现，低优先级）

- 位置：`internal/pkg/i18n`（`Negotiate` 语言协商 + 以"规范英文消息串"为键的译表 `LocalizeMessage`），接入 `handler.Error`。
- 语言来源：请求头 `X-Lang`（前端 `request.ts` 拦截器同步应用内语言切换）或 `Accept-Language`，默认 `en`（不改变既有 API 行为）。
- 价值：让后端**自给自足**，服务于**直接读 message 文本的非 Web API 消费者**（脚本/第三方集成）。对只被本前端调用的接口，加不加都行。
- 边界：仅覆盖**结构化错误响应**。`WithMessage` 自定义文案与未知串原样透传，**不会误译**。

## 后端唯一"非做不可"的 i18n（待立项）

真正只能在后端解决、且前端无法替代的，是**出站文案**：

- 飞书卡片 / 邮件通知正文
- AI / Agent 输出、巡检与报告正文
- 服务层散落的面向用户的 `fmt.Errorf("中文…")`（若它们会进入通知/卡片）

**难点**：这些要按**收件人**的语言偏好渲染（per-recipient locale），而不是按某个 HTTP 请求的 `Accept-Language`——因为一条告警可能同时推送给语言偏好不同的多个接收人。正确设计应：

1. 在用户/接收媒介上引入语言偏好字段；
2. 通知/卡片/报告渲染时按接收人偏好选择模板与文案；
3. 消息模板（`message_template`）支持多语言变体或按 locale 选择。

这是一套独立的设计，建议按高频模块（如通知管道、飞书卡片）分批推进，而非一次性外部化全部中文串。

---

## 决策小结

- 翻译**不应**整体搬到后端；前端 i18n 保留。
- API 错误用"码 + 前端翻译"已足够；后端错误译表作非 Web 消费者的兜底，属锦上添花。
- 后端 i18n 的投入重点应是**通知/卡片/AI 等出站内容的 per-recipient 本地化**——这才是后端 i18n 的正解。
